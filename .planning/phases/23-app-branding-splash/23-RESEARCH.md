# Phase 23: App Branding & Splash - Research

**Researched:** 2026-03-13
**Domain:** Flutter app icon generation, native splash screens, app name configuration
**Confidence:** HIGH

## Summary

Phase 23 covers three distinct concerns: (1) app launcher icon generation for Android and iOS, (2) app display name on the device home screen, and (3) a native splash screen that remains visible until household data is fully loaded.

The Flutter ecosystem has two purpose-built packages that are the canonical standard for this work: `flutter_launcher_icons` (dev-only code generator) and `flutter_native_splash` (runtime dependency + code generator). Both are Flutter Favorites, actively maintained, and resolve cleanly against the project's Dart 3.10.0 / Flutter 3.38.1 environment. Neither package is currently installed in this project.

Android app name is currently `android:label="valtra"` (lowercase) in `AndroidManifest.xml`. iOS `CFBundleDisplayName` is already `"Valtra"` (correct capital V) but `CFBundleName` is `"valtra"` (lowercase, should be fixed for consistency). The HouseholdProvider already exposes `isInitialized` and has an async `init()` that is awaited in `main()` before `runApp()` — but the first stream event from the database may arrive after `runApp`, meaning the households list is empty for one frame. The splash must stay visible until that first stream event populates the households list, not merely until `isInitialized == true`.

**Primary recommendation:** Add `flutter_launcher_icons ^0.14.4` as dev dependency; add `flutter_native_splash ^2.4.7` as a regular dependency; use `FlutterNativeSplash.preserve()` in `main()` and `FlutterNativeSplash.remove()` after the first household stream event, guarded by a listener on HouseholdProvider.

### Pattern 1: Icon Generation (flutter_launcher_icons)

**What:** A pubspec.yaml section drives a one-time code-gen command that writes all mipmap/AppIcon sizes.
**When to use:** Any time the icon source PNG changes.

**pubspec.yaml configuration:**
```yaml
# Source: https://pub.dev/packages/flutter_launcher_icons
dev_dependencies:
  flutter_launcher_icons: "^0.14.4"

flutter_launcher_icons:
  android: "launcher_icon"        # writes to res/mipmap-*/launcher_icon.png
  ios: true                        # writes to AppIcon.appiconset
  image_path: "assets/icon/icon.png"
  min_sdk_android: 21
  # Optional adaptive icon (Android 8+):
  # adaptive_icon_background: "#5F4A8B"   # Ultra Violet
  # adaptive_icon_foreground: "assets/icon/icon_fg.png"
```

**Run command:**
```bash
dart run flutter_launcher_icons
```

**Image requirements:**
- Source PNG should be 1024x1024 px (tool scales down automatically)
- iOS: image must fill the entire square — no transparent border padding
- Android adaptive icon foreground: tool adds 16% inset padding by default (`adaptive_icon_foreground_inset: 16`)
- If using a single `image_path` (not adaptive), the icon is used as-is for both legacy and adaptive Android

### Pattern 2: Native Splash Screen Generation (flutter_native_splash)

**What:** pubspec.yaml section drives code-gen; runtime API lets you hold the splash until app is ready.
**When to use:** Any Flutter project where first-frame flicker would be visible.

**pubspec.yaml configuration:**
```yaml
# Source: https://pub.dev/packages/flutter_native_splash
dependencies:
  flutter_native_splash: "^2.4.7"

flutter_native_splash:
  color: "#5F4A8B"              # Ultra Violet — light mode background
  color_dark: "#1A1A2E"         # dark background for dark mode
  image: assets/splash/splash_logo.png
  image_dark: assets/splash/splash_logo.png
  android_gravity: center
  ios_content_mode: center
  fullscreen: false             # keep status bar visible

  # Android 12+ SplashScreen API (mandatory separate section)
  android_12:
    color: "#5F4A8B"
    color_dark: "#1A1A2E"
    image: assets/splash/splash_logo.png   # 960x960 px recommended
    # Optional branding: assets/splash/branding.png  # 800x320 px
```

**Run command:**
```bash
dart run flutter_native_splash:create
```

**Undo command (if needed):**
```bash
dart run flutter_native_splash:remove
```

### Pattern 3: Keep Splash Until Households Loaded

**What:** Preserve the native splash in `main()`, then remove it after the HouseholdProvider receives its first stream result.
**When to use:** When async data must be available before showing the first frame.

**Critical timing insight:** In the current `main.dart`, `householdProvider.init()` is awaited before `runApp()`. However, `init()` sets `_isInitialized = true` synchronously *after* subscribing to the stream — the stream's first event (which populates `_households`) arrives asynchronously. Therefore, `isInitialized == true` does NOT mean the households list is populated. The splash must stay visible until `_households` is non-empty OR until at least one stream event has fired.

**Recommended approach — use a Completer-based wait in main():**

```dart
// Source: https://pub.dev/packages/flutter_native_splash
import 'package:flutter_native_splash/flutter_native_splash.dart';

void main() async {
  WidgetsBinding widgetsBinding = WidgetsFlutterBinding.ensureInitialized();
  FlutterNativeSplash.preserve(widgetsBinding: widgetsBinding);

  // ... all existing provider initialization ...
  await householdProvider.init();

  runApp(ValtraApp(
    // ... existing args ...
  ));

  // Remove splash after first household data is available.
  // HouseholdProvider.init() subscribes to the stream but the first
  // event arrives async. Wait for it before removing the splash.
  _removeSplashWhenReady(householdProvider);
}

void _removeSplashWhenReady(HouseholdProvider householdProvider) {
  // If init already received the first stream event, remove now.
  if (householdProvider.households.isNotEmpty ||
      householdProvider.isInitialized) {
    // Give one extra frame for the UI to render, then remove.
    WidgetsBinding.instance.addPostFrameCallback((_) {
      FlutterNativeSplash.remove();
    });
    return;
  }

  // Otherwise wait for the provider to notify (stream fires).
  void listener() {
    if (householdProvider.isInitialized) {
      householdProvider.removeListener(listener);
      WidgetsBinding.instance.addPostFrameCallback((_) {
        FlutterNativeSplash.remove();
      });
    }
  }
  householdProvider.addListener(listener);
}
```

**Simpler fallback** (acceptable if households always load within one frame after init):
```dart
void main() async {
  WidgetsBinding widgetsBinding = WidgetsFlutterBinding.ensureInitialized();
  FlutterNativeSplash.preserve(widgetsBinding: widgetsBinding);
  // ... init ...
  await householdProvider.init();
  runApp(ValtraApp(...));
  // Wait for first stream event via listener, then remove.
}
```

### Pattern 4: App Name Fix

**Android** — `android/app/src/main/AndroidManifest.xml`:
```xml
<application
    android:label="Valtra"   <!-- was "valtra" -->
```

**iOS** — `ios/Runner/Info.plist`:
```xml
<key>CFBundleName</key>
<string>Valtra</string>   <!-- was "valtra"; CFBundleDisplayName already correct -->
```

Note: `CFBundleDisplayName` (already "Valtra") is what appears on the device home screen on iOS. `CFBundleName` is the short internal name used in system menus. Both should be "Valtra".

### Anti-Patterns to Avoid

- **Using dev_dependencies for flutter_native_splash when using preserve/remove:** The runtime API requires it to be a regular `dependency`, not a dev dependency. Only use dev_dependencies if you never call `FlutterNativeSplash.preserve()`/`remove()`.
- **Calling FlutterNativeSplash.remove() before runApp():** The remove call must happen after the Flutter engine has rendered at least one frame. Use `addPostFrameCallback` or call it from within the widget tree.
- **Forgetting Android 12 section:** Android 12+ uses a completely different native splash mechanism (`SplashScreen` API). The `android_12:` block in `flutter_native_splash` is mandatory to avoid a white flash on Android 12+ devices.
- **Transparent padding on iOS icon:** Apple rejects icons with transparent borders. The source PNG must fill the 1024x1024 frame.
- **Not running the generator after changes:** Both `dart run flutter_launcher_icons` and `dart run flutter_native_splash:create` must be re-run whenever configuration or source assets change.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Resizing icon to 18 Android densities | Custom resize script | flutter_launcher_icons | Handles hdpi/xhdpi/xxhdpi/xxxhdpi, adaptive XML, round icons, Play Store 512px |
| Creating iOS AppIcon.appiconset | Manual Xcode drag-drop | flutter_launcher_icons | 17 sizes required; tool writes Contents.json correctly |
| Android 12 SplashScreen XML | Manual `res/values-v31/` XML | flutter_native_splash | Android 12 API is complex; tool handles windowSplashScreenBackground, windowSplashScreenAnimatedIcon, and theme |
| iOS LaunchScreen.storyboard | Manual Xcode editing | flutter_native_splash | Tool regenerates LaunchImage.imageset assets correctly |
| Keeping splash visible with a loading overlay | Flutter overlay widget | FlutterNativeSplash.preserve/remove | Flutter overlay appears *after* first frame — native splash is pixel-perfect during engine init |

**Key insight:** The "empty state flicker" on startup is caused by Flutter rendering its first frame (an empty widget tree) while data is still loading. No Flutter widget can prevent this — only the native splash mechanism (which exists before the engine starts) can cover this gap. `FlutterNativeSplash.preserve()` is the only correct solution.

## Common Pitfalls

### Pitfall 1: Android 12 White Flash
**What goes wrong:** App shows a white splash or wrong-colored splash on Android 12+ devices.
**Why it happens:** Android 12 introduced a mandatory new `SplashScreen` API. The old `windowBackground` drawable approach is ignored on API 31+.
**How to avoid:** Always include the `android_12:` section in `flutter_native_splash` configuration with matching colors.
**Warning signs:** Splash looks correct in emulator on API 29 but wrong on API 31+.

### Pitfall 2: Splash Removed Too Early
**What goes wrong:** Splash disappears and shows empty home screen for a frame before data loads.
**Why it happens:** `FlutterNativeSplash.remove()` is called at the wrong point — e.g., immediately after `householdProvider.init()` returns, but before the first stream event.
**How to avoid:** Remove splash only after `HouseholdProvider` has fired its first listener notification (stream result received), not just after `init()` completes.
**Warning signs:** Brief flicker of "No households" text before household name appears.

### Pitfall 3: Splash Never Removed
**What goes wrong:** App stays on splash screen forever.
**Why it happens:** `FlutterNativeSplash.remove()` is never called (e.g., exception before the call, or listener never fires).
**How to avoid:** Add a safety timeout or fallback removal. If `householdProvider.isInitialized` is true before the listener fires, remove immediately.
**Warning signs:** Stuck on splash during testing.

### Pitfall 4: Icon Has Transparent Background on iOS
**What goes wrong:** Icon looks broken or App Store submission fails.
**Why it happens:** iOS icons must not have any transparent areas.
**How to avoid:** Use a fully opaque 1024x1024 PNG; if the glassmorphism design has transparency, composite it onto the brand color (Ultra Violet #5F4A8B) background.
**Warning signs:** Icon looks correct in Flutter but Xcode shows a warning about transparency.

### Pitfall 5: App Name Not Updating on Device
**What goes wrong:** Device home screen still shows old name after build.
**Why it happens:** Android caches the app label; iOS sometimes caches too.
**How to avoid:** Uninstall and reinstall the app after changing `android:label` or `CFBundleName`. In CI, use a clean build.
**Warning signs:** Name shows correctly in build output but not on device.

### Pitfall 6: flutter_native_splash overwrites manually edited splash files
**What goes wrong:** Custom edits to `launch_background.xml` or `styles.xml` are lost on next `dart run flutter_native_splash:create`.
**Why it happens:** The generator regenerates these files completely.
**How to avoid:** All splash customization must go through `pubspec.yaml` flutter_native_splash config, not manual XML edits.

```dart

import 'package:flutter_native_splash/flutter_native_splash.dart';

void main() async {
  WidgetsBinding widgetsBinding = WidgetsFlutterBinding.ensureInitialized();
  FlutterNativeSplash.preserve(widgetsBinding: widgetsBinding);

  // ... existing init code unchanged ...
  final householdProvider = HouseholdProvider(HouseholdDao(database));
  await householdProvider.init();

  runApp(ValtraApp(/* existing args */));

  _removeSplashWhenReady(householdProvider);

void _removeSplashWhenReady(HouseholdProvider provider) {
  if (provider.isInitialized) {
    WidgetsBinding.instance.addPostFrameCallback((_) {

    if (provider.isInitialized) {
      provider.removeListener(listener);
      WidgetsBinding.instance.addPostFrameCallback((_) {
        FlutterNativeSplash.remove();

    }
  }
  provider.addListener(listener);
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual mipmap PNG copying | flutter_launcher_icons code-gen | ~2018 | Single source PNG, all sizes generated |
| windowBackground XML for splash | flutter_native_splash with android_12 block | Android 12 (2021), package v2.x | Must handle both old and new API |
| Calling remove() in initState | Using addPostFrameCallback | flutter_native_splash v2.3+ | Prevents "remove before render" crash |

**Deprecated/outdated:**
- Manual `launch_background.xml` editing: Overwritten by flutter_native_splash generator; all config must go through pubspec.yaml.
- `LaunchTheme` parent `@android:style/Theme.Light.NoTitleBar` for splash: Works but is supplemented by Android 12's `SplashScreen` API; both needed.

## Open Questions

1. **Glassmorphism icon asset availability**
   - What we know: BRAND-01 says "provided image asset" — the phase assumes a source PNG will be provided.
   - What's unclear: No asset file exists yet in the repo. Is it a flat icon or a glassmorphism effect that needs to be composited with a background for iOS?
   - Recommendation: Create a placeholder 1024x1024 PNG for testing; replace with final asset when provided. For iOS safety, the icon must have no transparency.

2. **Splash logo versus icon**
   - What we know: flutter_native_splash allows a separate `image` for the splash center logo.
   - What's unclear: Should the splash logo be the same as the app icon, or just the "house/gauge" mark without background?
   - Recommendation: Use the same source image for both initially. The planner can split them if the design requires.

3. **Testing FlutterNativeSplash in unit/widget tests**
   - What we know: The package does not document test behavior; `preserve()`/`remove()` are no-ops when the native splash infrastructure is not present (i.e., in test environments).
   - What's unclear: Whether calling `FlutterNativeSplash.remove()` in tests throws or silently passes.
   - Recommendation: Wrap `FlutterNativeSplash.remove()` behind a thin abstraction or verify it is safe to call unconditionally in test; test the `_removeSplashWhenReady` logic by mocking the provider, not the splash package.
