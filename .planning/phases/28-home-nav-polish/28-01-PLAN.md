    1. In `lib/database/tables.dart`, add a `personCount` column to the `Households` table class:
       ```dart
       IntColumn get personCount => integer()();
       ```
       This is a required field with NO default and NO nullable -- the context decision says "required field when creating a new household -- no default value." Place it after `description` and before `createdAt`.

    2. In `lib/database/app_database.dart`, bump `schemaVersion` from 3 to 4.

    3. In the `onUpgrade` migration, add a `from < 4` block:
       ```dart
       if (from < 4) {
         await customStatement(
           "ALTER TABLE households ADD COLUMN person_count INTEGER NOT NULL DEFAULT 1",
         );
       }
       ```
       The DEFAULT 1 is for the migration only (existing rows need a value). New rows will always have an explicit person_count from the form. The Drift table definition itself has no default -- this is correct because `integer()` without `.withDefault()` is required during insert.

       IMPORTANT: After adding the column via ALTER TABLE, existing rows get DEFAULT 1. New inserts via Drift will require the value explicitly because the Drift table definition has no `.withDefault()`. This is the desired behavior.

    4. Run `dart run build_runner build --delete-conflicting-outputs` to regenerate:
       - `lib/database/app_database.g.dart` (Household class gains `personCount` field)
       - `lib/database/daos/household_dao.g.dart` (mixin regenerated)

    5. Fix any compilation errors caused by the new required field. The generated `Household` class will now require `personCount` in its constructor. The generated `HouseholdsCompanion.insert` will require `personCount` as a positional/named parameter. Update ALL call sites:
       - `household_provider.dart` createHousehold and updateHousehold
       - Any test files that construct Household or HouseholdsCompanion

    DO NOT add `.withDefault()` or `.nullable()` to the table definition -- the form enforces the value.
